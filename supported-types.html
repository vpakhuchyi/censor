Supported Types

Censor supports a wide range of Go types and handles them recursively when possible. Here's a detailed overview of how different types are processed.

Structs​

By default, all struct fields are masked unless explicitly marked for display using the censor:"display" tag.

type User struct {
    ID       string `censor:"display"` // Will be displayed
    Name     string `censor:"display"` // Will be displayed
    Email    string                    // Will be masked
    Password string                    // Will be masked
}

user := User{
    ID:       "123",
    Name:     "John Doe",
    Email:    "john@example.com",
    Password: "secret123",
}

fmt.Println(censor.Format(user))
// Output: {ID: 123, Name: John Doe, Email: [CENSORED], Password: [CENSORED]}

Note: Unexported fields are always ignored.

Maps​

Maps are processed recursively, with both keys and values being formatted according to their respective types.

// Map with struct keys and slice values
type UserKey struct {
    ID      string `censor:"display"`
    Account string
}

data := map[UserKey][]string{
    {ID: "123", Account: "premium"}: {"email1@example.com", "email2@example.com"},
    {ID: "456", Account: "basic"}:   {"email3@example.com"},
}

fmt.Println(censor.Format(data))
// Output: map[{ID: 123, Account: [CENSORED]}: [[CENSORED], [CENSORED]], {ID: 456, Account: [CENSORED]}: [[CENSORED]]]

Map Type Display​

You can enable map type display in the output:

cfg := censor.Config{
    Encoder: censor.EncoderConfig{
        DisplayMapType: true,
    },
}

data := map[string]int{"one": 1, "two": 2}
fmt.Println(censor.Format(data))
// Output: map[string]int[one: 1, two: 2]

Slices and Arrays​

Slices and arrays are processed element by element, with each element formatted according to its type:

// Simple slice
numbers := []int{1, 2, 3}
fmt.Println(censor.Format(numbers))
// Output: [1, 2, 3]

// Nested slices
matrix := [][]int{{1, 2}, {3, 4}}
fmt.Println(censor.Format(matrix))
// Output: [[1, 2], [3, 4]]

// Slice of structs
users := []User{
    {ID: "123", Email: "user1@example.com"},
    {ID: "456", Email: "user2@example.com"},
}
fmt.Println(censor.Format(users))
// Output: [{ID: 123, Email: [CENSORED]}, {ID: 456, Email: [CENSORED]}]

Pointers​

Pointers are dereferenced and their underlying values are formatted according to the type rules:

// Pointer to struct
user := &User{ID: "123", Email: "john@example.com"}
fmt.Println(censor.Format(user))
// Output: {ID: 123, Email: [CENSORED]}

// Nil pointer
var nilUser *User
fmt.Println(censor.Format(nilUser))
// Output: nil

Basic Types​

Strings​

// Plain string
text := "Hello, World!"
fmt.Println(censor.Format(text))
// Output: Hello, World!

// String with sensitive pattern
email := "contact@example.com"
fmt.Println(censor.Format(email)) // With email pattern configured
// Output: [CENSORED]

Numbers​

// Integers
fmt.Println(censor.Format(42))          // Output: 42
fmt.Println(censor.Format(int64(999)))  // Output: 999

// Floating point
fmt.Println(censor.Format(3.14))        // Output: 3.14
fmt.Println(censor.Format(float32(1.5))) // Output: 1.5

Booleans​

fmt.Println(censor.Format(true))   // Output: true
fmt.Println(censor.Format(false))  // Output: false

Custom Types​

TextMarshaler Interface​

Types implementing the encoding.TextMarshaler interface are supported:

type CustomID string

func (id CustomID) MarshalText() ([]byte, error) {
    return []byte(fmt.Sprintf("ID-%s", string(id))), nil
}

id := CustomID("123")
fmt.Println(censor.Format(id))
// Output: ID-123

Interface Values​

Interface values are processed based on their concrete type:

var data interface{} = User{ID: "123", Email: "john@example.com"}
fmt.Println(censor.Format(data))
// Output: {ID: 123, Email: [CENSORED]}

Best Practices​

  1. Use the censor:"display" tag judiciously to control what information is visible
  2. Consider implementing encoding.TextMarshaler for custom types that need special formatting
  3. Be aware that unexported struct fields are ignored
  4. Test the output format with different combinations of nested types
  5. Use configuration options like DisplayMapType when additional type information is needed
